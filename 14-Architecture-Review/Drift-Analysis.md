# Drift Analysis & Consistency Checks

This document summarizes the cross-repository consistency checks performed during the discovery phase.

## Validation Results

### Application ↔ Helm
- **Service Names**: Matches perfectly. Names in `docker-compose.yml` (`auth-service`, `gateway`, etc.) match Helm sub-charts 1:1.
- **Ports**: Consistent. Internal routing maps `800x` target ports to K8s Service port `80`.
- **Environment Variables**: Consistent. Variables expected by FastAPI (`JWT_SECRET`, `INTERNAL_API_KEY`) are present in `Helm/values-common.yaml` or injected securely via Key Vault mapped in `flowforge-config`.

### Helm ↔ Terraform
- **Namespaces**: Matches. Terraform does not manage namespaces directly; ArgoCD maps `flowforge-dev` and `flowforge-prod` correctly to the deployed Helm charts.
- **Identity**: Matches perfectly. The `azure.workload.identity/client-id` injected in Helm uses the explicit object ID generated by Terraform.

### Terraform ↔ GitHub Actions
- **State Configuration**: Matches. `.github/workflows/terraform-gitops.yml` correctly targets `noelstf98` via OIDC.
- **Environment Names**: Matches. `Cloud-Track-dev` targets `dev` in both systems, and `main` targets `prod`.

## Detected Anomalies (Drift / Configuration Issues)
1. **Frontend Deployment Strategy**: The `frontend` Helm chart requests `strategy: blueGreen`. However, no deployment controller (like Argo Rollouts) is installed in the cluster to handle this. This setting will likely be ignored by Kubernetes, causing it to fall back to a `RollingUpdate`, meaning the intention of blue-green is not being realized.
